# 第二创新点：面向结构化审查节点的小模型下沉设计

## 1. 背景

当前系统 P4-BISG 已经实现了从用户测试意图到结构化 testcase 的自动生成流程。其核心链路包括：程序上下文构建、任务规约、多智能体协同生成、确定性校验与回退机制。系统中的不同 agent 分别承担意图分析、数据包生成、规则生成、规则审查和 oracle 预测等职责。

在现有设计中，所有 agent 默认都可由同类大模型承担。但从任务性质看，这些 agent 并不具有相同的复杂度和开放性：有的节点负责开放式生成，有的节点负责结构化审查，有的节点则负责解释与预测。因此，将所有节点统一交由高成本大模型处理，虽然实现简单，但并不一定是成本最优、结构最优的设计。

基于这一观察，本文提出第二个增强点：**根据 agent 的任务角色差异，引入异构模型分工机制，将开放式生成节点保留给大模型，将结构化审查节点下沉到小模型，在确定性 validator 的保护下实现成本与效果的平衡。**

在所有候选节点中，本文优先选择 `Agent5` 作为小模型替换对象。

## 2. 为什么选择 Agent5

### 2.1 Agent5 的任务边界最清晰

`Agent5` 的职责是对控制面规则生成结果进行审查。它的输入已经高度结构化，包括：

- `TaskSpec`
- 单场景 `packet_sequence`
- `RuleSetCandidate`

它的输出只有一个很小的结构化对象：

- `CriticResult`

这意味着 `Agent5` 的任务本质不是开放式内容生成，而是**对既有候选结果进行一致性判断与错误定位**。与需要从自然语言意图中推导任务约束的 `Agent1`、需要生成协议包序列的 `Agent2`、需要合成控制面实体的 `Agent4` 相比，`Agent5` 的任务范围更窄，输入输出形式也更加固定。

这种“输入结构强、输出结构小、职责单一”的节点，更适合作为小模型替换的第一目标。

### 2.2 Agent5 属于判别型节点，不属于核心开放生成节点

从系统工作流看，`Agent5` 不是负责“创造新内容”的节点，而是负责“判断已有内容是否合规”的节点。它的工作更接近一个结构化 critic，而不是一个开放式 generator。

这类任务通常具有几个特点：

- 需要识别局部错误，而不是规划全局方案；
- 更依赖规则对齐，而不是发散性推理；
- 更适合利用 prompt 约束输出为固定 JSON；
- 更容易通过现有样本和 validator 结果构造监督数据。

因此，从模型能力需求上看，`Agent5` 对大模型的依赖程度低于 `Agent1`、`Agent2` 和 `Agent4` 这类核心生成型节点。

### 2.3 Agent5 的错误风险最低，且下游有最强兜底

选择小模型时，不仅要考虑“哪个节点简单”，还要考虑“哪个节点换错后最不容易破坏整体系统”。从这一角度看，`Agent5` 是最安全的选择。

原因在于：`Agent5` 的输出并不是最终结论，它之后还会进入确定性校验阶段。系统当前已经提供了两层明确的 deterministic validator：

- `validate_control_plane_entities(...)`
- `validate_execution_sequence(...)`

也就是说，即使 `Agent5` 的判断存在偏差，系统后续仍会通过程序化校验对规则实体和执行顺序进行再次验证。若实体阶段整体失败，系统还可以进一步回退到最小可执行的 deterministic fallback 逻辑。

因此，`Agent5` 是一个**即使替换失败，也不容易立刻把主流程彻底破坏**的节点。这一点对于毕业论文中的创新验证尤其重要：它允许我们在控制风险的前提下，验证小模型下沉是否可行。

### 2.4 从论文论证角度，Agent5 最容易形成清晰结论

如果一开始就尝试替换 `Agent1` 或 `Agent2`，实验失败时很难判断是：

- 小模型不擅长理解自然语言意图；
- 小模型不擅长协议、拓扑和字段对齐；
- prompt 设计不足；
- 多阶段误差传播放大了早期偏差。

而选择 `Agent5` 则不同。由于它的任务边界清晰、输入输出结构固定、确定性 validator 强，因此实验结果的可解释性更强：

- 如果成功，可以说明：**结构化审查型节点确实可以由小模型承担**；
- 如果失败，也可以明确归因到审查质量、JSON 稳定性或复杂约束理解能力，而不是整个系统的多阶段传播问题。

这使得 `Agent5` 非常适合作为论文中“异构模型分工”创新点的首个验证对象。

## 3. 为什么选择 Qwen 小模型

在确定 `Agent5` 之后，接下来的问题是：应该使用哪一类小模型来承担这个审查型节点的任务。

本文优先考虑 Qwen 系列小模型，原因如下。

### 3.1 当前任务强依赖中文与结构化输出能力

本系统的输入并不只是纯英文技术说明，还混合了：

- 中文用户测试意图；
- 英文 prompt 契约；
- 程序上下文中的结构化字段；
- 严格的 JSON / schema 输出要求。

因此，候选小模型不仅要具备一般的 instruction following 能力，还应在以下方面表现稳定：

- 中文理解能力；
- 多语言混合上下文处理能力；
- 严格 JSON 输出能力；
- 对结构化字段的稳定对齐能力。

Qwen 系列模型在中文与多语言 instruction-following 场景下具有天然优势，也更符合本系统的输入特点。

### 3.2 Qwen2.5-7B-Instruct 适合作为启动模型

在实验初期，最重要的目标不是追求最小参数量，而是先确认“小模型承担 Agent5 是否可行”。为此，本文优先选择 `Qwen2.5-7B-Instruct` 作为首个启动模型，原因包括：

- 对结构化输出和 JSON 结果更稳；
- 对中文和中英混合上下文更友好；
- 参数规模仍属于中小模型范畴；
- 在 OpenAI-compatible 接口下更容易快速接入；
- 作为第一阶段验证模型，稳定性比极致压缩更重要。

换言之，`Qwen2.5-7B-Instruct` 并不是因为它一定是最终最优模型，而是因为它更适合作为“先打通小模型替换路线”的安全起点。

### 3.3 更小或更新代际的 Qwen 模型可作为后续增强方向

在完成 `Qwen2.5-7B-Instruct` 的接入与初步验证之后，可以进一步考虑使用更小或更新一代的 Qwen 模型，例如 `Qwen3` 或 `Qwen3.5` 系列，以进一步加强“小模型”叙事并降低推理成本。

但从实验顺序上，本文不将其作为第一步，而是作为第二阶段对比对象。这样做的原因是：

- 第一阶段需要尽可能排除接入链路、provider 配置和 prompt 稳定性带来的不确定性；
- 第二阶段才适合比较不同代际、不同参数规模的小模型在 `Agent5` 上的表现差异；
- 如果更小模型效果不足，再考虑通过 LoRA / QLoRA 方式微调，以提升其在结构化审查任务上的适配性。

因此，Qwen 系列在本文中的定位是分层的：

- `Qwen2.5-7B-Instruct`：用于启动验证；
- 更小或更新代际的 Qwen 模型：用于后续主实验或增强实验；
- 若零样本不足：再进入微调阶段。

### 3.4 Qwen 与现有工程接入路径兼容性较好

当前系统采用 `Agno + OpenAI-like` 的方式组织模型调用，而 Qwen 可通过兼容 OpenAI 的 API 形式接入，无需从根本上重写现有 agent 构造逻辑。这意味着：

- 小模型接入改动小；
- 可以在不改变 workflow 主体的情况下进行对比实验；
- 有利于将“创新点”集中在模型分工本身，而不是工程重构上。

对于毕业论文而言，这一点非常关键：它使实验能够围绕“模型角色分工是否合理”展开，而不是被部署细节和大规模系统修改干扰。

## 4. 本创新点的核心含义

综上，本文将第二创新点表述为：

> **面向结构化测试生成工作流的角色感知异构模型协同机制。**  
> 该机制根据不同 agent 的任务性质进行模型分层部署：开放式生成节点继续使用大模型，结构化审查节点下沉到小模型；同时借助既有确定性 validator 保证整体正确性，从而在不显著损害系统可用性的前提下降低推理成本，并提高系统的工程部署灵活性。

在这一机制中，`Agent5` 是最合适的首个小模型落地点，而 Qwen 系列小模型则是兼顾中文能力、结构化输出能力与工程可接入性的优先候选。

## 5. 当前阶段的实验定位

当前阶段的目标不是一次性完成所有小模型替换，而是先回答如下问题：

1. 小模型能否稳定承担 `Agent5` 的结构化审查任务？
2. 在 deterministic validator 的保护下，整体系统是否仍能保持可用？
3. 与原大模型方案相比，是否能够在推理成本或响应时间上获得收益？

若上述问题得到正向验证，则说明本文提出的“角色感知异构模型分工机制”是合理且有效的。后续再进一步扩展到更多模型或更多 agent，属于在此基础上的增强工作，而不是本创新点成立的必要前提。
