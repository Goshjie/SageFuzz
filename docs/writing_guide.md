# 硕士论文写作规范手册

> 本手册用于指导 AI 辅助撰写中文硕士学位论文。每次撰写任何章节内容前必须加载本文件。

---

## 一、禁用词表（AI 标记词）

以下词汇/短语在 AI 生成文本中高频出现，盲审专家一眼即可辨认。**全文禁止使用**，无例外。

### 1.1 连接词与过渡词

| 禁用         | 替代方案                               |
| ------------ | -------------------------------------- |
| 与此同时     | 同时（或直接用分号连接两个并列句）     |
| 具体而言     | 删掉，直接说具体内容                   |
| 需要强调的是 | 删掉，重要内容用句子本身的逻辑地位体现 |
| 特别地       | 其中 / 尤其 / 在...场景下              |
| 更进一步     | 进一步 / 在此基础上                    |
| 值得注意的是 | 删掉或改为"一个关键区别是"             |
| 总而言之     | 删掉，直接写结论                       |
| 综上所述     | 仅在本章小结中使用一次，其余场景删掉   |
| 不难发现     | 删掉                                   |
| 显而易见     | 删掉                                   |
| 毋庸置疑     | 删掉                                   |
| 事实上       | 删掉或改为"实际测试中"                 |
| 从本质上讲   | 删掉                                   |
| 在这一背景下 | 删掉或改为具体背景描述                 |
| 鉴于此       | 因此 / 为此                            |

### 1.2 修饰词与强调词

| 禁用              | 说明                                     |
| ----------------- | ---------------------------------------- |
| 核心优势          | 用"主要特点"或直接描述优势内容           |
| 显著提升/显著优于 | 给出具体数值，如"将 X 从 40% 提升到 78%" |
| 高效地/有效地     | 删掉，或用实验数据证明效率               |
| 至关重要          | 删掉，通过论证过程体现重要性             |
| 不可或缺          | 删掉                                     |
| 极大地            | 删掉或量化                               |
| 充分地            | 删掉                                     |
| 深入地            | 删掉                                     |

### 1.3 AI 高频句法模式（隐性 AI 味道）

以下模式不是单个词的问题，而是组合出现时暴露 AI 生成痕迹的句法习惯。每种模式单独出现一两次可以接受，但高频重复会立刻被识别为 AI 文本。

#### 1.3.1 破折号"——"滥用

**问题**：用"——"做插入解释，效果等同于括号（），每段都出现就成了 AI 指纹。

| 禁用模式                      | 替代方案                                                     |
| ----------------------------- | ------------------------------------------------------------ |
| "X——Y、Z、W——构成了..."       | "X 包括 Y、Z 与 W，共同构成了..."                            |
| "限定在某任务上——判断是否..." | "限定在某任务上：判断是否..." 或 "限定在某任务上，即判断是否..." |
| "面临困难——如果无法..."       | "面临困难。如果无法..." （断成两句）                         |
| "遵循准则——它倾向于..."       | "遵循准则，倾向于..."                                        |

全文"——"出现次数上限：每节不超过 2 处。超过时必须改为逗号、句号、冒号或重组句子。

#### 1.3.2 "这一"+抽象名词

**问题**："这一设计"、"这一问题"、"这一原则"、"这一压缩"、"这一提取过程" 是 AI 生成文本中极高频的指代模式。

| 禁用模式     | 替代方案                                    |
| ------------ | ------------------------------------------- |
| 这一设计     | 该设计 / 上述设计 / 该方案 / 直接用机制名称 |
| 这一问题     | 该问题 / 上述问题 / 直接描述问题内容        |
| 这一原则     | 该原则 / 上述原则                           |
| 这一提取过程 | 上述提取过程 / 整条管线                     |

全文"这一"出现次数上限：每节不超过 1 处。优先用"该"、"上述"、或直接用具体名词替代。

#### 1.3.3 "使/使得"连接

**问题**："使 LLM..."、"使得系统..."、"使模型..."——大量用"使"做因果连接是 AI 的典型习惯。

| 禁用模式          | 替代方案                             |
| ----------------- | ------------------------------------ |
| "X 使 Y 能够 Z"   | "Y 因此能够 Z" / "X 后，Y 得以 Z"    |
| "X 使得 Y 不会 Z" | "Y 因此不会 Z" / "X 避免了 Y 出现 Z" |
| "X，从而使 Y"     | "X，Y 因此..." / "X，Y 得以..."      |

全文"使/使得"做因果连接的上限：每节不超过 2 处。

#### 1.3.4 括号（）滥用

**问题**：用（）做插入解释或举例列举。常见变体包括：

- 解释性括号："（因为XXX）"、"（即XXX）"、"（含XXX）"
- 举例列举括号："（如 X、Y、Z 等）"、"（X、Y、Z 等）"、"（防范 X、Y 等）"——应将举例内容融入句子，如改为"X、Y 和 Z 等"或"如 X 和 Y"

术语缩写引入如"对象级授权破坏（BOLA）"、英文原名标注如"变质关系（Metamorphic Relations）"和节引用"（见 3.1 节）"属于合理使用，不计入此限制。

全文解释性（）上限：每节不超过 3 处。超过时将解释内容融入句子结构。

### 1.4 句式模板

| 禁用句式                                    | 问题            | 替代                                             |
| ------------------------------------------- | --------------- | ------------------------------------------------ |
| "基于 XX 的 YY" 作为标题                    | AI 论文标题模板 | 用更直接的表述                                   |
| "为了解决 XX 问题，本文提出了 YY 方法"      | 每段都这样开头  | 只在章首总述使用一次                             |
| "XX 具有以下特点：（1）...（2）...（3）..." | 列举堆砌        | 用自然段落叙述，除非列举项超过 4 项且每项较长    |
| "首先...其次...最后..."                     | 三段式模板      | 用逻辑关系词（"由于...因此..."、"在...之后..."） |
| "前述章节已经指出" / "如前文所述"           | 反复引用前文    | 直接说内容，必要时用"（见 3.1 节）"              |

### 1.5 频率控制词

以下词汇允许使用，但全章出现次数有上限：

| 词汇              | 第三章上限                      | 第四章上限 |
| ----------------- | ------------------------------- | ---------- |
| 本章              | 5 次                            | 5 次       |
| 本文              | 3 次                            | 3 次       |
| 本节              | 每节最多 1 次（用在节首总述段） |            |
| 传统方法/传统工具 | 3 次（仅在问题分析场景）        |            |
| 现有方案/现有方法 | 3 次                            |            |

## 二、学术写作规范

### 2.1 段落结构

每个自然段遵循以下结构（不必刻板，但要有意识）：

1. **主题句**：本段要说什么（一句话）
2. **展开**：具体内容、机制描述、分析推理
3. **收束**：本段结论或与下段的衔接

**禁止**：

- 一段只有一句话
- 一段超过 300 字（中文）而不分段
- 连续两段以相同句式开头

### 2.2 节内叙事结构

每一节（如 3.2.1、3.2.2）的内容应遵循学术论文的叙事逻辑：

```
问题陈述 → 现有做法的不足 → 本文的设计思路 → 具体机制 → 为什么这样设计（合理性论证）
```

**不是**：

```
我们做了 X → X 包含 A、B、C → A 的实现是... → B 的实现是...
```

前者是学术论文，后者是技术文档。区别在于：

- 学术论文：每个设计都有**动机**（为什么要这样做）和**论证**（为什么这样做比其他方式好）
- 技术文档：只描述"做了什么"和"怎么做的"

### 2.3 节首总述段

每个 ## 级节（如 3.2、3.3）的开头需要一段总述文字（3-5 句），说明：

- 本节要解决什么子问题
- 整体思路是什么
- 下面各小节的关系

这段文字**不需要**枚举所有小节标题（如"3.2.1 讲...，3.2.2 讲..."），而应该用自然语言描述逻辑关系。

**好的示例**：

> 黑盒场景下测试工具面对的是未文档化的原始流量，既不知道端点间的数据依赖关系，也无法直接获取请求参数的语义类型。本节从流量预处理出发，逐步构建从原始 HTTP 记录到结构化 API 知识的提取管线。

**差的示例**：

> 本节介绍黑盒 API 知识自动获取的三个子模块。3.2.1 节介绍流量归一化与接口建模，3.2.2 节介绍端点间依赖关系推断，3.2.3 节介绍运行时实体与参数收割。

### 2.4 图表引用

- 每张图/表在正文中**必须**被引用至少一次
- 引用格式："如图 3-1 所示"、"表 3-2 给出了..."
- 图表出现位置应紧跟首次引用的段落
- 图标题在图下方，表标题在表上方

### 2.5 术语一致性

全文同一概念只用一个术语，不要交替使用近义词：

| 统一用语 | 不要交替使用                                                 |
| -------- | ------------------------------------------------------------ |
| 外置记忆 | 外部记忆、外部存储、外部状态库                               |
| 依赖图谱 | 依赖图、依赖网络、关系图                                     |
| 请求渲染 | 请求组装、请求构建、报文生成                                 |
| 证据链   | 证据序列、审计链、追踪链                                     |
| 门控     | 拦截、阻断、过滤（在描述同一机制时）                         |
| 幻觉     | 臆造、编造、捏造（在描述 LLM 行为时）                        |
| 测试用例 | 探针、测试向量、payload（正文统一用"测试用例"，代码引用时可保留原名） |

### 2.6 数学符号与定义

- 核心概念在首次出现时必须给出定义（在 3.1 或 4.1 节）
- 定义格式：**定义 X.Y（名称）** + 形式化描述
- 后文引用定义时用"定义 X.Y"或直接使用术语名
- 同一符号全文含义一致，不要在不同节中复用同一字母表示不同含义

## 三、结构规范

### 3.1 章首总述

每章开头的引导段落（在第一个 ## 节之前）应包含：

1. 本章要解决的核心问题（1-2 句）
2. 为什么这个问题难（传统方法不足 + LLM 直接用的失败，2-3 句）
3. 本章的核心思路（1 句）
4. 本章的结构概览（用自然语言，不要编号列举节标题）

总长度控制在 150-250 字。

### 3.2 标题规范

- **章标题**：体现创新点 + 问题域，如"上下文卸载的业务逻辑漏洞测试生成"
- **节标题（##）**：体现子问题，如"面向 LLM 的状态外置与上下文控制"
- **小节标题（###）**：体现具体方案，如"可执行外置记忆机制"

标题中**禁止**：

- 冒号分割式："XXX：一种基于 YYY 的方法"
- 过度堆砌："基于 XX 与 YY 的 ZZ 驱动 WW 机制研究"
- 动词开头："探索..."、"研究..."
- 纯名词堆叠超过 4 个

### 3.3 本章小结

每章最后一节的本章小结应：

1. 用 2-3 句话回顾本章解决了什么问题
2. 总结核心设计思想
3. 简要提及实验结论（如有）
4. 用 1 句话引出下一章

总长度控制在 100-150 字。**不要**逐节复述内容。

## 四、常见问题清单

### 4.1 ❌ 功能堆砌

**问题**：按系统模块逐一介绍功能，每段写一个模块的"是什么→怎么做的"。
**修正**：先提出问题，再分析为什么现有做法不够，最后引出设计。

### 4.2 ❌ 列举代替论述

**问题**：大量使用编号列表介绍系统特性，如"（1）支持多身份 （2）自动解析依赖 （3）智能变异"。
**修正**：用自然段落叙述，逻辑关系用句子结构体现。仅在列举项多于 4 项且每项需要较长说明时使用编号列表。

### 4.3 ❌ 五要素并列

**问题**：连续罗列恰好 5 个并列要素（AI 生成文本的典型特征）。
**修正**：如果确实有 5 个要素，考虑合并或拆分，避免恰好 5 个的排列。

### 4.4 ❌ 自我吹捧

**问题**："本方法显著优于现有方案"、"实验结果充分证明了本方法的有效性"。
**修正**：让数据说话——"在 X 数据集上，本方法的 Y 指标为 Z%，高于基线方法的 W%"。

### 4.5 ❌ 空洞的"创新点"声明

**问题**：在引言或小结中说"本文的主要创新点包括：（1）提出了... （2）设计了... （3）实现了..."。
**修正**：在摘要和引言中简明概括，不在每章重复声明。

### 4.6 ❌ 过度引用"本章/本节/本文"

**问题**：每段都以"本节..."开头。
**修正**：用具体的技术术语代替，如"外置记忆机制"代替"本节提出的方法"。

### 4.7 ❌ 翻译腔

**问题**：直接从英文思维翻译，如"在这个上下文中"（in this context）、"出于这个原因"（for this reason）。
**修正**：用中文自然表达——"在这种场景下"、"因此"。

## 五、自检清单

每完成一节内容后，按以下清单逐项检查：

- [ ] 禁用词表中的词是否出现？（全文搜索）
- [ ] 频率控制词是否超限？
- [ ] **正文中是否出现函数名、变量名、类名、字段名？**（最重要，必须全部替换为抽象描述）
- [ ] 是否存在代码调用流程的描述？（"先调用 A，再调用 B"）
- [ ] 每段是否有主题句？是否超过 300 字？
- [ ] 是否存在连续两段以相同句式开头？
- [ ] 每个设计是否有动机（为什么）和论证（为什么这样好）？
- [ ] 是否存在功能堆砌（按模块逐一介绍而不是按问题组织）？
- [ ] 编号列表是否必要？能否改为自然段落？
- [ ] 是否恰好出现 5 项并列？
- [ ] 所有术语是否前后一致？
- [ ] 图表是否都在正文中被引用？
- [ ] 本节开头是否有合适的总述段？
- [ ] 每段是否有洞察（insight）或方法论意义的总结？
- [ ] 整体读起来像学术论文还是像产品文档？

## 六、代码与论文的关系（最重要的一条）

### 6.1 核心原则：论文是学术作品，代码只是参考素材

代码创新点清单和源码文件是**素材来源**，不是论文内容本身。写论文时，绝对不能写成代码的文字翻译或功能文档。论文的目标是**提出方法、论证合理性、展示学术贡献**，代码只是验证这些贡献的载体之一。

### 6.2 学术提升：从工程实现到方法论

同一件事在代码和论文中的表述层次完全不同：

| 代码里的样子                                  | 论文里应该怎么写                                             |
| --------------------------------------------- | ------------------------------------------------------------ |
| `if not rh_ids: reject`                       | "引入基于证据存在性的结论生成前置约束，将'先有证据后有结论'从规范性建议转化为系统级的强制不变量" |
| `lookup_reference_traffic()` 返回历史成功请求 | "提出示例驱动的请求修复策略：当生成请求触发服务端错误时，系统从历史成功交互中检索结构相似的请求作为修复参照，将 LLM 的自由猜测收敛为有据可依的局部调整" |
| `DualEngineScheduler` 根据复杂度选引擎        | "设计基于复杂度感知的混合生成框架：对结构简单的端点采用确定性规则变异保障稳定性，对语义复杂的端点引入 LLM 语义变异扩展覆盖面，两者通过可解释的路由策略动态切换" |
| `ContextVar` + `reset_rh_accumulator`         | "构建基于执行闭包的证据隔离机制，确保并发场景下每个检测实例维持独立的证据链，从架构层面消除证据交叉污染" |
| `P3a: majority vote on status code`           | "提出基于协议层信号的声明-执行一致性校验：以实际 HTTP 响应状态码作为 ground truth，对 LLM 声称的状态码进行交叉验证并自动纠正不一致项" |

关键区别：

- **代码**说的是"做了什么"（what）和"怎么做的"（how）
- **论文**要说的是"为什么要做"（why）、"解决了什么问题"（problem）、"比现有方法好在哪里"（contribution）、"背后的道理是什么"（rationale/insight）

### 6.3 合理的学术包装技巧

论文允许且应该对工程实现进行学术提升，这不是吹嘘，而是**揭示设计背后的方法论意义**：

**1) 给机制起学术名称**

不要直接用代码里的函数名或变量名，而要提炼出抽象概念：

| 代码命名                                        | 学术名称                 |
| ----------------------------------------------- | ------------------------ |
| `save_request_history` + `rh_id_collector_hook` | 证据链自动注入机制       |
| `no-evidence gate` + `P3a` + `P3b`              | 多层反幻觉门控           |
| `lookup_reference_traffic`                      | 示例驱动的请求修复       |
| `scan_context` + `resolve_scan_context`         | 上下文依赖注入           |
| `_compute_complexity` + `DualEngineScheduler`   | 复杂度感知的混合生成策略 |

**2) 提炼设计原则（insight）**

每个重要设计背后应该有一个可以用一句话概括的洞察：

- 证据链绑定的洞察：**"结论的可信度取决于其是否可追溯到原始观测"**
- 上下文卸载的洞察：**"LLM 的可靠性边界在于语义推理，而非状态记忆"**
- 硬门控的洞察：**"对于安全关键任务，提示词约束（软约束）不足以替代程序化校验（硬约束）"**
- 错误语义反馈的洞察：**"失败响应不应被视为噪声丢弃，而应被解析为可操作的修复信号"**

这些洞察应该出现在每节的开头或设计论证段落中，让审稿人感受到"这不只是工程优化，而是有思想的方法设计"。

**3) 建立与学术概念的关联**

把工程实现关联到已有的学术概念上，提升理论深度：

| 工程实现                 | 可关联的学术概念                                             |
| ------------------------ | ------------------------------------------------------------ |
| 流量清洗→聚类→中心点抽样 | 信息瓶颈（information bottleneck）、主动学习中的代表性采样   |
| 证据锚点回注             | 不变量保持（invariant preservation）、后处理保真             |
| scan_context 依赖注入    | 关注点分离（separation of concerns）、最小权限原则           |
| 熔断器 + 自适应并发      | 闭环控制（closed-loop control）、优雅降级（graceful degradation） |
| 确定性 Workflow 编排     | 控制流与数据流分离、确定性有限状态机                         |
| 双引擎变异               | 探索-利用权衡（exploration-exploitation trade-off）          |

**4) 用对比突出贡献**

论文中每个设计都应该有一个（隐式或显式的）对比对象，说明"如果不这样做会怎样"或"现有做法为什么不够"：

- "直接让 LLM 管理测试状态 → 上下文膨胀、状态遗忘" → 所以要外置
- "依赖提示词要求 LLM 诚实输出 → 软约束无法保障" → 所以要硬门控
- "通用压缩模型不理解证据语义 → 关键字段被丢弃" → 所以要证据锚点回注
- "固定使用 LLM 或规则 → 要么不稳定要么覆盖不足" → 所以要复杂度路由

### 6.4 代码细节的抽象层次（极其重要）

论文中**绝对禁止**出现以下内容：

- 具体的变量名、函数名、类名（如 `scan_id`、`build_scan_context`、`validate_strict_evidence_context`）
- 具体的字段名罗列（如"包含 project_id、scan_id、execution_id、identity_id 和 auth_mode 五个必填字段"）
- 具体的数据格式（如 ObjectId、JSON Schema、ContextVar）
- 具体的框架 API（如 Agno 的 RunContext、dependencies 字典）
- 代码级的优先级链（如"RunContext → dependencies → 显式参数的三级优先级"）

这些内容属于实现细节，审稿人不需要知道你的变量叫什么名字。论文要描述的是**机制的设计思想和方法论意义**。

**反面示例**（代码文档式写法，❌）：

> SDA-Fuzzer 将扫描上下文从 LLM 提示词中剥离，转由系统层的依赖注入机制管理。该机制的核心是 scan_context——一个包含 project_id、scan_id、execution_id、identity_id 和 auth_mode 五个必填字段的结构化上下文对象。在扫描启动时，编排器通过 `build_scan_context` 构造该对象，并经由 `build_scan_context_dependencies` 将其封装为 Agno 框架的 dependencies 字典，注入到每个 agent 的运行时上下文（RunContext）中。当工具函数被调用时，`resolve_scan_context` 按照"RunContext → dependencies → 显式参数"的三级优先级自动解析上下文。

**正面示例**（学术论文式写法，✓）：

> 传统的 LLM 驱动测试方案将任务标识、操作身份、认证模式等运行时上下文全部编码在提示词中，由模型在每次工具调用时显式传递。随着调用链深度增长，模型对这些上下文的记忆逐渐衰减，尤其在并发场景下容易发生身份漂移——一个本应以特定身份执行的检测实例在经过多轮压缩后可能"遗忘"自身角色。
>
> 针对这一问题，所提机制采用依赖注入的方式将运行时上下文从模型的认知负担中完全剥离。系统在测试启动时构造结构化的上下文描述符，通过框架级的依赖传播机制自动注入到每个检测实例的执行环境中。工具函数在被调用时从执行环境中自动获取所需上下文，而非从模型输出中解析。这一设计将上下文传递的正确性从"模型是否记得传参"转变为"系统是否完成注入"，消除了模型记忆衰减对上下文一致性的影响。
>
> 在此基础上，系统对上下文描述符的合法性施加严格约束：身份标识与认证模式之间的语义一致性在每次工具执行前被自动校验，语义冲突的请求（如匿名身份搭配强制认证）不被执行，而是转化为带有原因标注的可解释跳过记录。这种"宁可跳过也不错执行"的策略将上下文漂移从隐式的结果失真转化为显式的可统计事件。

注意两者的区别：

- 反面示例在讲**代码怎么实现的**——函数名、参数名、优先级链
- 正面示例在讲**方法怎么设计的**——动机、思路、原则、效果

**转化规则**：当你想写一个实现细节时，问自己"审稿人需要知道这个吗？"如果答案是"不需要，他只需要知道这一步在做什么、为什么要做"，就提升一个抽象层次。

| 想写的代码细节                                              | 应该提升为                         |
| ----------------------------------------------------------- | ---------------------------------- |
| scan_context 包含 5 个字段                                  | "结构化的上下文描述符"             |
| build_scan_context / resolve_scan_context                   | "系统在启动时构造上下文并自动注入" |
| RunContext → dependencies → 显式参数                        | "多级回退的自动解析机制"           |
| validate_strict_evidence_context                            | "语义一致性校验"                   |
| identity_id ∈ {anonymous, user_a, user_b, admin}            | "身份标识属于预定义集合"           |
| ContextVar + reset_rh_accumulator                           | "基于执行闭包的隔离机制"           |
| ObjectId regex + DB existence check                         | "格式校验与存在性验证的双重防伪"   |
| _truncate_response_body / _RH_RESPONSE_BODY_MAX_BYTES=256KB | "响应体按安全阈值截断"             |

### 6.5 禁止的写法

即使要做学术提升，也不能过度：

- ❌ 生造不存在的学术术语（"基于量子纠缠的证据绑定"）
- ❌ 把简单实现吹成重大理论突破（一个 if 判断不需要写成"定理"）
- ❌ 在没有实验数据支撑时说"显著提升"
- ❌ 关联不相关的学术概念来装深度
- ❌ 每段都强调"创新性"——让论证过程自然体现创新，不需要反复声明
- ❌ 在正文中出现任何函数名、变量名、类名、具体字段名
- ❌ 罗列系统的内部参数或配置项
- ❌ 描述代码的调用顺序或执行流程（"先调用 A，再调用 B，B 返回后传给 C"）

### 6.6 自检：代码味 vs 学术味

写完一段后问自己：

- 这段话里有没有**函数名、变量名、字段名**？→ 如果有，全部替换为抽象描述
- 如果把这段话拿给一个不看代码的审稿人，他能理解**为什么要这样设计**吗？→ 如果不能，缺动机和论证
- 如果把这段话拿给一个写代码的工程师，他会觉得这就是**技术文档**吗？→ 如果是，缺学术提升
- 这段话里有**洞察**（insight）吗？→ 如果没有，补一句概括性的方法论总结
- 读者看完这段会觉得"这是一个**方法贡献**"还是"这是一个**功能描述**"？→ 目标是前者

## 七、语气与文风

### 7.1 目标文风

中文计算机领域硕士论文的典型文风：**客观、平实、精确**。

- 不要热情洋溢（"这是一个令人兴奋的发现"）
- 不要过于谦虚（"本文仅做了初步尝试"——除非在局限性讨论中）
- 不要口语化（"搞定了"、"干掉了"）
- 不要文学化（"在数据的海洋中遨游"）

### 7.2 人称与视角

- 使用"本文"、"所提机制"、"系统"等第三人称表述
- 避免"我们"（部分学校允许，但保守起见不用）
- 禁止"我"
- 禁止"你"（读者称为"读者"或不直接称呼）

### 7.3 陈述方式

- **事实陈述**用陈述句："该机制将状态码校验前置到数据写入层。"
- **设计理由**用因果句："由于 LLM 可能在同一轮对话中预写状态码，系统在保存结果前从实际执行记录中提取真实状态码进行交叉校验。"
- **对比**用对比句："不同于依赖提示词约束的软方法，该机制在数据写入路径上实施硬校验。"

## 八、结构性 AI 指纹检测清单

> 以下模式在第一章 AI 味道深度审查中被发现并修复。与第一章 1.3 节的词级/句法级模式不同，这些是**段落级和篇章级**的结构性 AI 指纹，更隐蔽也更难用简单搜索发现。每次写完一节后应逐项检查。

### 8.1 恰好 N 项并列（N=4 或 5）

**问题**：AI 生成文本倾向于把并列事物组织成恰好 4 或 5 项的"A、B、C 和 D"结构。单独一处不明显，但一篇文章中反复出现（≥3 处）就成为强 AI 指纹。

**检测方法**：全文搜索"、.*、.*和"或"、.*、.*与"模式，人工判断是否恰好 4 项。

**修复策略**：

- 合并为 3 项（将两个相近的合并）
- 拆成 2+2（前半句说两个，后半句说两个）
- 改为自然散文叙述，不用顿号列举
- 如果确实需要 4 项以上，改为流水线式叙述："先做 A，再经 B 处理，随后 C，最终 D"

**全文上限**：恰好 4 项并列不超过 2 处/章，恰好 5 项并列不超过 1 处/章。

### 8.2 模板句式的段落级重复

**问题**：连续段落使用完全相同的句式骨架，最常见的形式是：

- "针对 XX 问题，所提方法..." × 连续 2 段
- "第一，XX。...第二，XX。...第三，XX。" 三段平行
- "X 是第一条路径。...Y 是第二条路径。...Z 是第三条路径。" 机械编号

**检测方法**：检查连续 2–3 段的开头 10 字是否使用相同句式框架。

**修复策略**：

- 只保留第一处使用该骨架，后续段落换不同的句式切入
- 对于"第一/第二/第三"，改为以内容本身开头，用自然过渡连接
- 对于路径式分类，各段用不同的逻辑起点（"最直接的思路是...""当 X 不可得时...""Y 为此开辟了新路线"）

### 8.3 "围绕/聚焦/面向" 范围设定词过密

**问题**："围绕""聚焦""面向"都是合法的学术用语，但 AI 文本倾向于在每段切换话题时都用一个范围设定词开头。

**全文上限**：三者合计每章不超过 3 处。优先用具体的技术主语开头代替。

### 8.4 "以...为..." 形式结构过密

**问题**："以 X 为核心思路""以 X 为基础""以 X 为中心"等文言化结构本身无问题，但同一节内密集出现（≥3 处）会产生刻板感。

**全文上限**：每节不超过 2 处。

### 8.5 "将 X 从 Y 剥离/转化为/收窄/限定" 转化动词过密

**问题**：AI 在表达"A 替代 B"的语义时，倾向于反复调用"将 X 从 Y 中剥离""将 X 限定为 Y""将 X 收窄到 Y"等同一句式骨架。

**全文上限**：每节不超过 2 处。同一"将...从..."骨架在全章不超过 3 处。

### 8.6 超长单句（>100 中文字符）

**问题**：硕士论文中超过 100 个中文字符的单句（句号到句号之间）节奏感差，审稿人容易失去耐心。AI 倾向于在一个句子中塞入所有信息。

**修复策略**：以分号或句号拆分，保持每句 40–80 字。

### 8.7 演变叙事三连击

**问题**：1.1 首段典型 AI 开场模式——"随着 X 的普及，Y 已从 A 演化为 B，逐渐成为 C"。在一句话内同时使用"已从...演化为""已成为""逐渐"等多个演变标记。

**修复策略**：用具体事实开头代替宏大叙事，保留一个演变标记即可。

### 8.8 "的"字定语链堆叠

**问题**：同一句中出现 4 个以上"的"字的定语链，如"系统间**的**通信通道演化为业务能力**的**直接承载体与组织边界**的**主要暴露面**的**基础能力"。

**修复策略**：拆成两句，或将部分定语改为独立从句。每句中"的"字不超过 3 个。

### 8.9 总结句尾模式过密

**问题**："...构成了""...提供了""...形成了""共同反映出""共同指向"等总结性句尾，单独使用无问题，但同一节内出现 3+ 次就成为 AI 语感。

**全文上限**：同一总结句尾动词每节不超过 1 次，不同总结句尾动词合计每节不超过 2 次。

### 8.10 高频术语集中

**问题**：某个术语在全章出现次数过多（如"机制"18 次），说明缺乏同义替换意识。

**参考上限**（每章）：

- "机制"≤8 次（可用"方法""方案""策略""手段"分散）
- "系统"≤15 次（可用"工具""框架""方案"分散）
- 其他高频术语参照此标准，同一术语不应在连续两句中出现

### 8.11 绪论章中的内部章节引用

**问题**：在第一章（绪论）的"研究内容与创新点"等节中出现"（3.2 节）""（4.3 节）"形式的括号内章节引用。绪论是概述性章节，读者尚未阅读后续内容，插入精确到小节级别的引用显得机械且不符合阅读习惯。

**AI 成因**：AI 把章节编号当作"精确性"的体现，认为提及一个概念时附上其所在小节号能增加可信度。但在绪论中，读者对后续章节的编号体系尚无认知，"3.2 节"对他来说只是一个无意义的数字标签。人类写作者在绪论中谈及后续内容时只会用章级粒度（"第三章"）或内容描述（"生成侧的设计"），不会精确到小节。

**适用范围**：仅限第一章。第三至五章的正文中"（见 3.1 节）""如 4.2 节所述"等引用属于正常的前后文衔接。

**修复策略**：

- 用"第三章""第四章"等章级引用代替小节级引用
- 或直接用内容描述代替引用，如"在生成侧的设计中"代替"（3.3 节）"
- 允许在 1.4 节（论文组织结构）中使用"第三章...""第四章..."的章级概述

### 8.12 "研究内容与创新点"节的叙事顺序

**问题**：AI 倾向于先罗列系统的功能模块或章节贡献，再附带提及对应的问题。学术论文的惯例是先说问题、再说贡献——读者需要先理解"为什么要做"才能评价"做了什么"的价值。

**AI 成因**：AI 的生成过程是从系统描述出发向外扩展，"我做了 X"比"X 为什么需要做"更容易直接生成。人类写作者则从读者视角反推——读者必须先认同问题的存在才会关心解法。

**正确顺序**：问题描述 → 解决思路 → 具体贡献 → 创新点总结

**错误顺序**：系统架构概述 → 各章功能描述 → 顺带提及问题

**修复策略**：

- 开头 1–2 段用于问题陈述，说明现有技术面临的困难
- 中间段落按"问题→方案"的逻辑分组，而非按章节顺序平铺
- 创新点放在最后，作为前述工作的自然归纳

### 8.13 括号内信息压缩（AI 信息焦虑）

**问题**：AI 倾向于在一个句子中尽可能多地塞入信息，把本应在后文展开的细节用括号压缩插入当前位置。常见形式包括：

- **括号内章节标注**："在测试生成方面（第三章）"——把 1.4 节（论文组织结构）的导航信息渗透进 1.3 节（研究内容）
- **括号内技术词列举**："API 知识获取（流量归一化、依赖图谱构建、运行时实体收割）"——在概述层面插入子模块清单
- **括号内实验编号**："消融实验（G0→G4）"——读者尚未建立对该编号体系的认知，符号串没有语义
- **括号内对等解释**："时间轴（操作前后状态对比）、空间轴（跨端点信息关联）"——每个概念后面都跟一个括号解释

**AI 成因**：AI 缺乏"读者此刻知道多少"的建模能力，无法判断当前位置适合哪个粒度的信息。它把所有层次的细节平铺在同一个句子里，用括号做层次区分。人类写作者对信息的分层有直觉——绪论只需要概念级描述，子模块清单和实验编号留给正文章节。

**检测方法**：统计每段中括号（）的数量。解释性括号（排除术语缩写引入和合理节引用）超过 2 处/段即需审查。

**修复策略**：

- **章节标注**：删除。1.3 节不需要告诉读者"这是第几章的内容"，那是 1.4 的职责
- **技术词列举**：展开为自然语言的流水线叙述，如"先从流量中提炼接口模型，再构建端点间依赖图谱"
- **实验编号**：用语言描述实验做了什么，如"从基线逐步叠加各层组件"，编号留到实验节首次引入
- **对等解释**：改为冒号引导的分述句，或拆成独立短句逐一说明

**全文上限**：绪论章中解释性括号每段不超过 1 处；正文章节每段不超过 2 处。

### 8.14 同章节内的显式回指

**问题**：AI 在切换话题时习惯加入"如 X.Y 节所述""前文已指出"等显式回指，将其当作学术文风的标志。在同一章内部（尤其是绪论这种线性阅读的章节），上一节的内容就在前一页，读者不需要导航提示。

**AI 成因**：AI 的训练数据中学术论文频繁使用跨章回指（"如第二章所述"），这在跨章场景下确实合理。但 AI 无法区分跨章引用和章内引用的场景差异，把跨章的衔接习惯照搬到了相邻两节之间。

**适用范围**：

- **禁止**：同一章内相邻节之间的回指（"如 1.2 节所述"出现在 1.3 开头）
- **允许**：跨章引用（第四章引用"如 3.1 节定义的..."）
- **允许**：同章但间隔较远的引用（1.4 引用 1.1 提出的概念，中间隔了 1.2 和 1.3）

**修复策略**：直接删除回指，让内容自然衔接。如果确实需要语义过渡，用内容本身做衔接（"这两个层面的困难..."）而非节号。

### 8.15 "就 X 而言...就 Y 而言..."对称二分总结

**问题**：AI 在综述或总结段落中，倾向于用完全对称的框架把两个主题并排摆放："就测试生成而言，...就漏洞判决而言，..."。两段使用相同的句法起手，读起来像填表。

**AI 成因**：AI 在处理"两方面"内容时，默认生成对称结构——两段长度相近、句式相同、起手完全一致。这是因为 Transformer 在续写第二段时，注意力权重集中在第一段的句法骨架上，最省力的生成方式就是复制骨架换内容。人类写作者会刻意打破对称，用不同的句式切入第二个主题。

**修复策略**：

- 第一主题保持正常起手，第二主题换一个不同的句法入口
- 例如："在测试生成侧，长链路状态探索...；在漏洞判决侧，预言机问题与 LLM 幻觉构成双重制约——..."用句式差异和破折号打破对称
- 或将两个主题合并为一段连续论述，用分号而非段落分隔

### 8.16 文献综述中的"沿 N 个方向展开"结构预告

**问题**：AI 在组织文献综述时，倾向于先宣布数量再分段展开："预言机研究沿三个方向展开"。这种"先数数再说内容"的写法读起来像目录，不像叙述。

**AI 成因**：AI 在生成结构化内容时需要先规划后续段落的数量，"沿 N 个方向展开"实际上是 AI 对自身后续生成计划的外化。人类写作者通常直接进入第一个方向的讨论，读者会在阅读过程中自然感知到有几条路线，不需要提前被告知。

**修复策略**：

- 删除数量预告，直接用第一个方向的内容起头
- 例如将"沿三个方向展开。最直接的思路是..."改为"在 API 安全场景中，最直观的做法是..."
- 如果确实需要过渡，用内容层面的衔接："除了从规范中挖掘约束，灰盒可观测性增强提供了另一种思路"

### 8.17 综述尾段的刻意扣题

**问题**：AI 在文献综述（1.2）的最后一段结尾处加入"本研究即从这两个方向给出回应"之类的句子，把综述的 gap 和自己的工作显式连接。这种"扣题"在 1.3（研究内容）中才是合适的位置；在 1.2 尾部出现，使综述失去了客观评述的独立性。

**AI 成因**：AI 的训练数据中，大量学术论文在 Related Work 结尾处确实会写一句"本文从...角度出发"做过渡。但绪论中 1.2 → 1.3 是紧邻的两节，gap 的陈述在 1.2 末尾完成即可，不需要在同一段里把解法也预告出来。

**修复策略**：

- 综述尾段只陈述 gap，不提及本研究的回应
- "本研究如何回应"留给 1.3 的第一段来说

### 8.18 空洞过渡句（"本研究从上述 N 个层面给出回应"）

**问题**：AI 在问题陈述段与解法段之间插入一句空洞的过渡，如"本研究提出的 X 系统从上述两个层面分别给出回应"。这类句子不携带任何新信息——读者已经知道前面在说问题，后面会说解法，不需要一句话显式宣布"接下来我要回应了"。

**AI 成因**：AI 生成长文本时，内部的"规划器"会在话题切换处插入元叙述（meta-narrative）来维持上下文连贯。"从上述两个层面分别给出回应"本质上是 AI 在对自己说"好，问题说完了，现在开始说解法"，这个内部状态转换被外化成了一个句子。人类写作者直接进入解法描述即可，话题切换由段落和句意完成，不需要一句话做"播报"。

**常见变体**：

- "本研究/本文针对上述问题提出了 X"（1.3 开头已有"本研究针对..."的章级概述，段内不需要重复）
- "为了解决上述挑战，所提系统..."（"为了解决"是 AI 最高频的过渡模板之一）
- "为此，本文设计了..."（当"为此"前面紧跟的就是问题描述时，因果关系已经隐含在段落结构中）
- "基本观察是：..."/"核心观察是：..."（洞察播报——先宣布"我要说 insight 了"再说 insight 本身。直接陈述观察内容即可，不需要用冒号引导的标签句式。尤其注意连续段落重复使用"基本观察是"会构成 8.2 的模板句式重复）
- "概括而言，本研究的贡献在于：..."（总结播报——"贡献在于"后面跟冒号再展开，是 AI 最典型的结尾模板。改为直接陈述效果，如"上述工作的整体效果是"或直接以具体内容开头）
- "其'成功表象'会系统性地误导..."（引号概念化——把一个可以直说的意思包装成引号术语，再接"会系统性地..."等修饰。改为直接描述现象，如"越权访问在协议层面看起来完全正常"）
- "...仍然悬而未决"（文学化收束——"悬而未决"是成语化表达，在学术论文中显得不自然。改为"尚未得到有效解决""仍是开放问题"等平实表述）

**修复策略**：

- 直接删除过渡句，让解法描述以系统名或核心概念开头
- 例如将"本研究提出的 SDA-Fuzzer 系统从上述两个层面分别给出回应。SDA-Fuzzer 在生成侧采用..."合并为"SDA-Fuzzer 在生成侧采用..."
- 如果删除后两段之间确实显得跳跃，用一个承载实质内容的句子做过渡，而非空洞的元叙述

### 8.19 文献综述中的系统性桥接段（小节级前向引用）

**问题**：AI 在撰写文献综述（第二章）时，倾向于在每个小节末尾插入一段"桥接段"，将该小节讨论的技术概念精确映射到后续章节的具体小节。例如："第三章的运行时实体收割机制（3.2.3 节）通过...""第四章的权限轴差分（4.2.3 节）以..."。当这种模式在每个小节都出现时（第二章中曾多达 17 处），综述不再是客观的技术脉络梳理，而变成了"本文方案的预告片"。

**AI 成因**：AI 在生成综述内容时，内部维护了一个"所有概念→本文对应方案"的映射表。每当讨论完一个背景概念，AI 会自动触发映射查找并生成桥接段，这是一种"关联焦虑"——担心读者无法建立背景与贡献的联系。人类写作者在综述中只偶尔提及"后续章节将进一步讨论"，且通常只用章级引用（"第三章""第四章"），不会在每个小节都精确到 X.Y.Z 节。

**检测方法**：

- 在综述章中搜索"第三章""第四章"及"X.Y 节""X.Y.Z 节"形式的引用
- 统计每小节末段是否以前向引用收束
- 如果超过 50% 的小节末尾都有映射到后续章节的段落，即构成系统性桥接

**修复策略**：

- **大多数桥接段应直接删除**，综述的职责是客观梳理现有工作和研究缺口，不是预告本文方案
- 如果小节末尾需要收束，用研究缺口或开放问题的陈述代替桥接，如"在纯黑盒条件下，该问题仍是一个开放挑战"
- 允许在本章小结（2.4）中用 1-2 句话以章级粒度概括后续方向，但不应精确到小节
- 正文章节（第三、四、五章）中引用文献综述的概念时可以反向引用"如 2.1 节所述"，这是合理的前后文衔接

**全文上限**：综述章中前向引用到后续章节的段落不超过 3 处/章，且只使用章级引用（"第三章""第四章"），禁止小节级引用（"3.2.3 节""4.2.1 节"）。
