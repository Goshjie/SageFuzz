# 3.4 节图占位与绘制说明

本文档用于说明 `3.4 面向网络行为的 P4 意图感知与种子生成方法设计` 中各图的占位位置、推荐图名以及具体绘制方式，目的是让后续人工绘图或大模型绘图都能够直接按照文字描述完成图示。以下说明以论文表达为主，并结合本系统已经验证过的场景给出可直接放入图中的文字内容。

---

## 图3-3 P4-BISG 系统总体架构与模块交互关系

### 图名
图3-3 P4-BISG 系统总体架构与模块交互关系

### 放置位置
对应小节：`3.4.1 系统总体架构与模块交互`

### 建议图形类型
采用“分层架构图 + 数据流箭头”的形式，而不是简单线性流程图。建议把整张图分成四层：输入层、上下文层、生成与调度层、输出层。每一层内部再细分为多个小模块。这样可以同时体现系统组成、模块职责以及模块之间的信息流动关系。

### 画布布局建议
- 画布横向放置，比例建议为 `16:9`
- 自上而下分为四层，每层内部从左到右排列小模块
- 层与层之间用粗实线箭头表示主数据流
- 同层内部模块之间用细实线箭头表示调用关系
- 审查、重试、回退相关路径使用虚线箭头表示反馈流
- 建议在右上角放一个小图例：
  - 实线箭头：主流程
  - 虚线箭头：审查/回退反馈
  - 双向虚线：共享上下文访问

### 四层结构与具体模块

#### 第一层：输入层
这一层建议放 3 个并列模块。

1. `用户完整意图输入`
框内文字建议：
- 功能/策略描述
- 拓扑与角色约束
- 观察目标
- 测试目标类型

2. `程序输入文件`
框内文字建议：
- BMv2 JSON
- DOT 图
- P4Info
- Topology
- P4 Source

3. `运行配置`
框内文字建议：
- model config
- retry/timeout
- memory config
- output path

#### 第二层：程序上下文层
这一层建议画成“ProgramContext 构建中心 + 六个上下文子模块”结构。

中心模块：`ProgramContext 构建器`
框内文字建议：
- 统一程序上下文
- 结构化索引
- 查询接口注册

围绕中心模块的六个子模块：
1. `解析器上下文`
- 协议路径
- transition 条件
- header 定义

2. `控制流上下文`
- 表跳转关系
- 表深度排序
- 路径约束

3. `控制平面上下文`
- 表结构
- 匹配键
- 动作签名

4. `状态对象上下文`
- register
- counter
- meter

5. `拓扑上下文`
- host 信息
- link 信息
- role 绑定依据

6. `源码证据上下文`
- 源码检索
- 片段查询
- 补充语义证据

这六个模块都用箭头汇入 `ProgramContext 构建器`，再由 `ProgramContext 构建器` 向下输出到生成与调度层。

#### 第三层：生成与调度层
这一层是图3-3的重点，建议拆成“工作流调度模块”和“多智能体协同模块”两部分。

##### （A）工作流调度模块
建议画成左侧一个纵向调度框，分成以下 5 个小块：
1. `意图采集与测试目标选择`
2. `任务规约循环`
3. `场景生成循环`
4. `规则生成控制`
5. `结果汇总与落盘`

##### （B）多智能体协同模块
建议画成右侧 6 个智能体框，并按阶段排列：

1. `Agent1 语义分析`
- 意图分类
- 角色绑定
- 任务规约输出

2. `Agent3 任务审查`
- 语义完整性检查
- 缺失约束反馈

3. `Agent2 数据包生成`
- packet_sequence
- 场景报文步骤

4. `Agent3 序列审查`
- 协议栈检查
- 字段关系检查

5. `Agent4 规则生成`
- entities
- control_plane_sequence
- execution_sequence

6. `Agent5 规则审查`
- 表项合法性
- forbidden_tables 检查

7. `Agent6 Oracle 预测`
- expected_outcome
- expected_rx_host
- processing_decision

建议把这部分画成两列：
- 左列：Agent1、Agent2、Agent4、Agent6（生成者）
- 右列：Agent3、Agent3、Agent5（审查者）

其中：
- `Agent1 -> Agent3(任务审查)`
- `Agent2 -> Agent3(序列审查)`
- `Agent4 -> Agent5`
- `Agent5 -> Agent6`

同时从工作流调度模块指向各智能体，表示由调度器发起调用。

#### 第四层：结果校验与输出层
这一层建议放 4 个模块。

1. `参数修复与 schema 纠正`
框内文字建议：
- JSON repair
- 参数别名归一化
- 类型纠正
- required 检查

2. `确定性校验`
框内文字建议：
- 场景约束检查
- 协议字段一致性
- 角色绑定一致性
- oracle 对齐检查

3. `回退生成`
框内文字建议：
- packet fallback
- entity fallback
- oracle fallback

4. `最终输出`
框内文字建议：
- testcase JSON
- run index
- timing 信息
- case records

### 箭头关系说明
建议至少画出以下箭头：

#### 主流程实线箭头
- `用户完整意图输入 -> 意图采集与测试目标选择`
- `程序输入文件 -> ProgramContext 构建器`
- `运行配置 -> 工作流调度模块`
- `ProgramContext 构建器 -> 工作流调度模块`
- `工作流调度模块 -> Agent1`
- `Agent1 -> Agent3(任务审查)`
- `Agent3(任务审查) -> 工作流调度模块`
- `工作流调度模块 -> Agent2`
- `Agent2 -> Agent3(序列审查)`
- `Agent3(序列审查) -> 工作流调度模块`
- `工作流调度模块 -> Agent4`
- `Agent4 -> Agent5`
- `Agent5 -> 工作流调度模块`
- `工作流调度模块 -> Agent6`
- `Agent6 -> 结果校验与输出层`
- `结果校验与输出层 -> 最终输出`

#### 虚线反馈箭头
- `Agent3(任务审查) --反馈--> Agent1`
- `Agent3(序列审查) --反馈--> Agent2`
- `Agent5 --反馈--> Agent4`
- `确定性校验 --失败--> 回退生成`
- `回退生成 --结果--> 最终输出`

#### 共享上下文访问
从 `ProgramContext 构建器` 出发，用虚线或细线指向所有智能体框，表示各智能体共享统一程序上下文与工具查询能力。

### 框内建议文字风格
- 尽量使用短语，不写长句
- 每个框控制在 2 至 4 行
- 结构化对象名可以保留英文，例如：
  - `TaskSpec`
  - `packet_sequence`
  - `entities`
  - `oracle_prediction`

### 图中可直接使用的总说明语句
如果需要在图中央或图下注释中放一句概括性文字，可以使用：

`P4-BISG 以程序上下文为约束基础，以任务规约为中间语义层，通过多智能体协同生成和一致性校验，将用户测试意图转换为可执行 testcase。`

### 推荐出图重点
图3-3不应只表现“先做什么、后做什么”，而应同时表现三件事：
1. 系统由哪些小模块构成；
2. 模块之间怎样交互；
3. 审查、修复和回退如何嵌入主流程。

换言之，这张图应该更接近“系统架构图”，而不是普通流程图。

### 图3-3 缩略版说明（用于第三章正文）

如果第三章正文不希望出现过于复杂的系统架构图，可以在图3-3中使用缩略版。缩略版仍保留系统的分层结构、主要模块以及关键的顺序关系，但不在矩形框内放置说明性小字，只保留每个模块的加粗标题。这样做的目的，是在不损失整体方法链路的前提下，降低图面信息密度，使读者先建立整体认识，再通过正文理解细节。

#### 缩略版推荐图名
图3-3 P4-BISG 系统总体架构与模块交互关系

#### 缩略版适用场景
- 第三章正文首次介绍方法总体结构时使用
- 需要先让读者把握系统层级与主流程，而不是立刻进入子模块细节时使用
- 后续若还会给出更细的结构图，则图3-3宜采用缩略版

#### 缩略版保留的模块
缩略版仍建议保留以下模块标题：

第一层：输入层
- 用户意图输入
- 程序输入文件
- 运行配置

第二层：程序上下文层
- ProgramContext 构建器
- 解析器上下文
- 控制流上下文
- 控制平面上下文
- 状态对象上下文
- 拓扑上下文
- 源码证据上下文

第三层：生成与调度层
- 工作流调度
- Agent1 语义分析
- Agent3 任务审查
- Agent2 数据包生成
- Agent3 序列审查
- Agent4 规则生成
- Agent5 规则审查
- Agent6 Oracle 预测

第四层：结果输出层
- 参数修复与校验
- 回退生成
- testcase 输出

#### 缩略版顺序关系要求
缩略版必须体现“调度器负责组织阶段顺序，但智能体执行并非并行启动”。因此图中不宜从“工作流调度”同时发出多条平行箭头分别指向 Agent1、Agent2 和 Agent6，否则容易造成所有智能体并行工作的误解。推荐的画法是：

1. 工作流调度只向生成链路的入口发出一条主箭头；
2. 智能体之间按照阶段顺序串行排列；
3. 任务审查和序列审查用虚线箭头反馈到前一阶段；
4. 规则生成、规则审查和 oracle 预测继续沿主链路顺序展开；
5. 最终由生成链路汇入结果校验与输出层。

也就是说，在缩略版中应强调“调度控制是统一的，但生成过程是分阶段推进的”，而不是把所有智能体画成并行分支。

#### 缩略版箭头保留原则
缩略版不必保留全部细粒度箭头，但建议保留以下几类主关系：

1. 输入汇聚关系
- 输入层汇入 ProgramContext 构建器和工作流调度

2. 上下文支撑关系
- ProgramContext 构建器向生成链路提供共享约束

3. 阶段式主流程关系
- 工作流调度进入生成链路入口
- Agent1 语义分析 -> Agent3 任务审查
- Agent2 数据包生成 -> Agent3 序列审查
- Agent4 规则生成 -> Agent5 规则审查 -> Agent6 Oracle 预测
- 生成链路 -> 结果输出层

4. 审查反馈关系
- Agent3 任务审查反馈回 Agent1
- Agent3 序列审查反馈回 Agent2
- Agent5 规则审查反馈回 Agent4

5. 回退关系
- 参数修复与校验失败后指向回退生成
- 回退生成再指向 testcase 输出

#### 缩略版框内文字要求
- 每个矩形框内只保留模块标题
- 标题使用加粗字体
- 不再列出模块内部条目，例如不写“协议路径、表深度排序、路径约束”等小字
- 箭头上的说明文字也应减少，只保留必要的“反馈”“校验”“输出”等关键词

#### 缩略版图面效果要求
- 图面应明显比完整版更疏朗
- 保持四层结构不变
- 保持主要模块不缺失
- 保持系统总体链路清晰可读
- 读者应能够一眼看出：输入经过上下文建模、任务调度和阶段式多智能体生成，最终形成 testcase 输出

#### 正文配套说明建议
若正文中采用缩略版图3-3，可在图前后配套使用如下表述：

“图3-3给出了 P4-BISG 的总体架构。该图主要展示系统的分层结构和模块间主交互关系。为避免图面过密，图中仅保留了主要模块及阶段顺序，各模块内部职责将在后续小节中分别展开。”

这样可以避免读者期待在一张图中读到全部细节，也能使缩略图与正文自然衔接。

## 统一绘图建议

1. 使用黑白或灰蓝配色，避免过于鲜艳。
2. 统一使用矩形框和箭头，保证风格一致。
3. 图中文字以中文说明为主，仅保留少量结构化对象名，如 `TaskSpec`、`packet_sequence`、`execution_sequence`。
4. 每张图尽量控制在 5 至 8 个主框之间，避免信息过密。
5. 图下注释只说明符号或缩写，不展开代码实现细节。
