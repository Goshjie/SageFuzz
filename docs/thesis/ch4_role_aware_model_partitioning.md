# 第四章 角色感知的异构模型分工与小模型下沉

本章讨论的问题是：在多智能体测试种子生成流程已经建立的前提下，是否有必要让所有节点继续使用同一类大模型，以及不同角色节点是否可以采用不同规模的模型完成协同生成。该问题的难点不在于单次调用是否成功，而在于当模型被替换后，整条工作流是否还能保持结构稳定、语义一致和工程可用。围绕这一点，本章提出一种面向角色异质性的模型分层部署方法，并通过多组对照实验分析结构化审查节点的小模型可行性与全小模型方案的局限。

## 4.1 引言

第三章已经完成了面向网络行为的 P4 测试种子生成方法设计，并证明该方法能够围绕用户意图生成包含数据包序列、控制平面规则和 oracle 的结构化 testcase。随着系统进入可运行阶段，一个新的问题随之出现：在包含多个 agent 的协同工作流中，是否所有节点都需要使用同一类大模型。若继续沿用统一模型方案，系统部署成本和调用时延会随链路长度持续增长；若贸然将所有节点同时替换为小模型，又可能破坏任务规约、数据包生成和规则组织的稳定性。

本章不把“小模型替换”视为孤立的工程优化，而是把它视为工作流层面的角色分工问题。不同 agent 的职责并不相同，有的节点负责开放式生成，有的节点负责结构化审查，有的节点负责结果解释。模型规模是否需要下沉，取决于该节点承担的任务边界、错误传播范围以及下游是否存在确定性校验。本章因此引入角色感知的异构模型分工思路，目标是在不明显破坏主流程的前提下降低系统推理成本。

## 4.2 研究背景

多智能体协同生成系统常把复杂任务拆成多个语义子问题，再由不同 agent 分别完成。该设计降低了单个模型同时处理多类约束的难度，也让中间结果具备更清晰的对象边界。P4-BISG 的六个 agent 正是在这一框架下组织起来的：`Agent1` 负责意图规约，`Agent2` 负责数据包生成，`Agent3` 负责约束审查，`Agent4` 负责控制面规则生成，`Agent5` 负责实体规则审查，`Agent6` 负责 oracle 预测。

这六个节点在任务性质上并不对称。`Agent1`、`Agent2` 和 `Agent4` 需要从不完整约束出发生成新的结构化对象，输出空间较大，错误一旦出现，往往会向后续阶段传播。`Agent3` 和 `Agent5` 则更接近审查型节点，输入已经具有较强结构，输出通常是较短的判断结果或反馈信息。`Agent6` 虽然也输出结构化结果，但更偏向解释与预期规划。若继续采用统一大模型方案，系统可以保持相对简单的部署方式；若要引入小模型，下沉位置就不能只看参数量，而要看该节点对生成自由度、错误容忍度和下游兜底强度的要求。

当前开源模型与托管模型的生态也为这种分工提供了现实条件。一方面，Qwen 系列在中文、多语言和结构化输出任务上具备较强适配性，可以作为本系统的候选小模型。另一方面，托管平台与开源权重的可用性并不完全一致，同一模型族在不同入口下的支持情况也可能不同。模型分层部署不仅是算法或 prompt 的选择问题，还包含 provider 兼容性、接口稳定性和任务适配性。该背景决定了本章的实验必须同时关注“模型是否能输出结果”和“结果是否足以支撑该节点职责”这两个层面。

## 4.3 问题描述

本章关注的不是“某个小模型是否偶尔能生成一条 JSON”，而是如下问题：给定已经稳定运行的多智能体测试种子生成流程，如何在不破坏主流程语义完整性的前提下，将一部分节点从大模型切换到小模型，并判断该切换是否合理。形式上，可将该问题表述为：在一组 agent 节点集合 $A=\{a_1,\dots,a_n\}$ 上，为每个节点分配模型规模 $m_i$，使系统在保证 testcase 结构与行为语义可接受的前提下，减少整体推理成本与延迟。

这一问题至少包含三个子约束。第一，节点的任务边界必须清晰。如果某一节点本身承担的是开放式规约或多步生成，其错误后果会向后传播，那么该节点不宜作为首个小模型下沉对象。第二，切换后的节点输出必须可校验。若下游不存在确定性 validator，仅凭模型自检难以判断结果是否可接受。第三，实验评价标准不能停留在“跑通”层面，还需要比较通过率、重试次数、fallback 触发情况以及输出是否真正符合 schema 与任务语义。

围绕这三个约束，本章将六个节点按角色分成两类：一类是开放式生成节点，包括 `Agent1`、`Agent2` 和 `Agent4`；另一类是结构化审查节点，包括 `Agent3` 与 `Agent5`。其中，`Agent3` 同时承担任务规约审查和数据包序列审查，职责混合度较高；`Agent5` 只负责控制面规则与执行顺序的结构化审查，输入边界更稳定，输出形式也更简单。基于这一差异，本章优先选择 `Agent5` 作为首个小模型落点，并以“全小模型统一替换”作为对照方案。

## 4.4 角色感知的异构模型分工方法

本章提出的方法可称为**角色感知的异构模型分工方法**。它不追求一次性压缩整条工作流，而是先分析每个节点的任务属性，再决定该节点是否适合下沉到小模型。方法的核心思想是：开放式生成节点继续使用大模型，结构化审查节点优先尝试小模型，再由确定性 validator 负责兜底。

该方法包含四个步骤。第一步是节点角色分析。对每个 agent，分别考察其输入结构程度、输出自由度、错误传播范围和下游校验强度。第二步是候选节点筛选。只有当节点输出边界清晰、输出类型规模小、错误后果局部且有确定性校验支撑时，才进入小模型候选集合。第三步是分层部署。保持 `Agent1`、`Agent2`、`Agent4` 等开放式生成节点继续使用原有大模型，把 `Agent5` 作为结构化审查节点切换到小模型，并保留原工作流中的 deterministic validator、fallback 与日志记录。第四步是对照验证。除“仅 `Agent5` 使用小模型”的主方案外，再额外设置“全部节点统一切换小模型”的对照组，用于检验角色分工是否真的必要。

选择 `Agent5` 的原因集中体现在三点。其一，`Agent5` 的输入由 `TaskSpec`、单场景 `packet_sequence` 与 `RuleSetCandidate` 组成，结构边界明确，输出只有 `CriticResult`，任务本质更接近判别而不是生成。其二，`Agent5` 的错误传播范围较小，其输出之后仍会经过 `validate_control_plane_entities` 与 `validate_execution_sequence` 两层确定性校验。其三，即使实体阶段失败，系统仍可退回最小可执行实体构造逻辑，这让该节点具备较高的试错安全性。基于这三个特点，把 `Agent5` 作为首个下沉对象，比直接替换 `Agent1` 或 `Agent2` 更有可解释性，也更容易形成稳定实验结论。

本章在模型选择上采用两级策略。一级是可行性验证模型，用于判断小模型能否承担 `Agent5` 的基本结构化审查任务；二级是更激进的压缩候选模型，用于进一步检验更小参数规模在当前 provider 下的可用性。前一类实验选用 `qwen2.5-7b-instruct` 作为启动模型，其目的在于先确认下沉方案本身成立。后一类实验选用 `qwen3-4b`，其目的在于测试更小模型在当前托管接口下是否仍能承担同类任务。两级模型的定位不同：前者用于支撑方法成立，后者用于验证压缩边界。

## 4.5 实验分析

本节采用三组实验支撑上述方法。第一组实验是整条工作流的自动化对照。第二组实验是保留真实 agent 输入与 prompt 的节点级回放。第三组实验则进一步将 `Agent5` 从完整工具栈中剥离，只保留真实 prompt 与真实输入，验证其结构化审查能力。实验过程中所用的代表任务为 Stateful Firewall、Link Monitor 和 Fast Reroute，分别对应状态策略验证、监测型任务和带外部动作的路径切换任务。

| Stateful Firewall | 全大模型（qwen-plus） | PASS | PASS | 1 | 2 | 0 |

### 4.5.2 真实节点回放结果

为排除全流程中上游失败对下游节点结论的遮蔽，本章进一步使用历史真实输入做 agent 级回放。实验脚本为 `scripts/replay_point2_agent_samples.py`，结果保存在 `runs/point2_agent_replays_2026-03-09T152746Z`。该实验保留了真实 agent prompt、真实输入样本和真实 schema 约束，对 `Agent1`、`Agent2`、`Agent4`、`Agent5` 分别做小模型回放。

| Link Monitor | invalid_output | invalid_output | invalid_output | invalid_output |

早期回放中，失败样本大量表现为 `"Connection error."`。在排除网络限制后重新运行，`qwen3-4b` 的真实 agent 回放结果保存在 `runs/point2_agent_replays_2026-03-10T061209Z`，`qwen2.5-7b-instruct` 的对应结果保存在 `runs/point2_agent_replays_2026-03-10T061545Z`。重跑后可以看到，两类小模型都表现出“局部节点可用、全节点不稳”的共同特征：`Agent2` 在 `firewall` 场景上能够返回合法结构化结果，`Agent5` 在 `qwen3-4b` 下也能在 `firewall` 与 `link_monitor` 两个任务上成功回放；但 `Agent1` 与 `Agent4` 仍然频繁输出不满足 schema 的结果。由此可以看出，问题并不只是早期网络波动，而是节点角色与模型能力之间存在明显不匹配。

### 4.5.3 Agent5 的单节点可行性与模型选择

为了验证 `Agent5` 的角色本身是否适合小模型，本章分别对 `qwen2.5-7b-instruct` 与 `qwen3-4b` 做了 prompt-only 回放。该实验只保留真实 `Agent5` prompt 和真实历史输入，不引入完整工具栈。`qwen2.5-7b-instruct` 的稳定结果保存在 `runs/point2_agent5_prompt_only_2026-03-10T050648Z`，`qwen3-4b` 的对应结果保存在 `runs/point2_agent5_prompt_only_2026-03-10T061210Z`。

对 `qwen2.5-7b-instruct` 而言，结果如下：

- Stateful Firewall：成功输出合法 `CriticResult`，结论为 PASS；
- Link Monitor：成功输出合法 `CriticResult`，结论为 PASS；
- Fast Reroute：成功输出合法 `CriticResult`，结论为 PASS。

对 `qwen3-4b` 而言，结果同样全部成功：

- Stateful Firewall：成功输出合法 `CriticResult`，结论为 PASS；
- Link Monitor：成功输出合法 `CriticResult`，结论为 PASS；
- Fast Reroute：成功输出合法 `CriticResult`，结论为 PASS。

这说明在剥离完整工具栈之后，`Agent5` 这一结构化审查节点本身确实适合小模型下沉。与此同时，两类模型在 prompt-only 条件下的 wall-clock 时间都维持在数秒级：`qwen2.5-7b-instruct` 大致分布在 `3.39 s` 到 `5.20 s`，`qwen3-4b` 大致分布在 `2.48 s` 到 `3.02 s`。这种速度差异来自模型规模与推理复杂度的差别，但并不意味着更快的模型一定更适合整条链路。对当前系统而言，更重要的是：一旦恢复真实 agent 构造方式，`Agent1` 与 `Agent4` 这类开放式生成节点仍然不稳，而 `Agent5` 的局部可用性可以被稳定观察到。因此，`Agent5` 适合作为首个小模型落点，这一点不受模型型号变化影响。

### 4.5.4 firewall 场景的重复实验

`qwen3-4b` 在 `firewall / all_small` 上曾出现过一次单次 `PASS`。为了判断这一结果是否具有稳定性，本章进一步在可联网环境下对 `firewall` 任务做了 2 轮重复对照，结果保存在 `runs/point2_firewall_repeats_2026-03-10T065438Z`。重复实验包含三组配置：全大模型、仅 `Agent5` 小模型、全小模型。

统计结果表明：

- 全大模型：2 轮中 1 轮 `PASS`，1 轮失败；
- 仅 `Agent5` 小模型：2 轮全部失败；
- 全小模型：2 轮全部失败。

按 wall-clock 时间统计，全大模型两轮平均约为 `91.82 s`，仅 `Agent5` 小模型约为 `16.16 s`，全小模型约为 `16.18 s`。其中两类小模型配置都在十几秒内提前退出，说明其较短运行时间来自前期失败，而不是高效完成完整 testcase 生成。这个重复实验进一步表明，先前出现过的单次 `all_small` 成功不具有稳定可复现性，不能作为“全小模型方案成立”的依据。

### 4.5.5 小结

本节实验支持以下结论。第一，不能把“所有节点统一切换小模型”视为默认方案，因为在当前系统和当前 provider 条件下，该方案在复杂任务上并不稳定。第二，`Agent5` 作为结构化审查节点，确实具备小模型下沉的合理性；这一点已经由 `qwen2.5-7b-instruct` 的单节点回放结果支撑。第三，模型选择不能只看参数规模，还要看 provider 兼容性和 schema 输出稳定性。对当前系统而言，角色感知的异构模型分工是成立的，但其中的小模型候选需要继续在可用性更稳定的型号上推进，而不能简单把更小模型直接接入整条链路。
