# 3.3 问题定义与设计目标

本章关注的问题是：在 P4 程序测试中，如何将用户给出的自然语言测试意图自动转化为可执行的测试种子，并使该种子能够服务于后续行为验证、缺陷触发和结果比对。这里的“测试意图”并不是简单的功能描述，而是面向网络行为的测试需求，例如某类通信是否被允许、某种状态是否会在特定输入下发生转移、某条链路的监测结果是否能够被正确观测、或者在特定环境变化下转发路径是否按预期切换。与传统仅围绕数据平面路径或单个输入包进行测试生成的方法不同，本文所处理的问题具有更强的场景性：测试目标不仅依赖报文内容，还依赖主机角色、发送顺序、控制平面配置以及必要的外部动作。

从形式上看，本文考虑的输入是用户给出的完整测试意图与目标程序上下文。其中，测试意图描述待测行为、角色约束和观察要求；程序上下文包括编译产物、控制流结构和拓扑信息。输出则不是单一报文，而是一组结构化测试种子。该种子至少应包含数据包序列、必要的控制平面规则、执行顺序以及预期结果描述。由此，本文将问题定义为：给定用户测试意图 $I$ 和程序上下文 $C$，自动生成结构化测试种子 $T$，使得 $T$ 能够在目标程序上驱动与 $I$ 对应的行为验证过程，并支持后续结果判断。

从测试语义上看，这一问题至少包含四项约束。第一，生成结果应与用户意图在语义上保持一致，不能仅因为某条路径可达就构造与测试目标无关的输入。第二，生成结果应与程序上下文保持一致，即所使用的协议栈、主机绑定、表项结构和动作签名必须与目标程序真实结构匹配。第三，生成结果应具备执行语义，不能只输出若干孤立输入，而应能够反映测试中的顺序关系、环境动作和观察点。第四，生成结果应具备一定稳定性，即在模型输出存在轻微偏差时，系统仍能够通过修复、校验和回退机制输出可用 testcase。

上述问题之所以难，主要来自三个方面。首先，用户测试意图通常处于较高语义层，而 P4 程序的可执行输入处于较低语义层，两者之间存在明显的语义鸿沟。用户通常关心“谁与谁通信”“何时允许回包”“链路断开后是否切换”，但并不会直接给出精确的头字段、表项匹配键或状态读取方式。其次，P4 程序的行为往往依赖多种异构因素共同作用，包括解析器路径、控制平面规则、主机角色和外部环境事件，仅靠单一报文往往无法构成有效测试。最后，大语言模型虽然适合处理自然语言意图，但其输出存在结构不稳定、字段漂移和工具调用误差等问题，若缺乏约束与校验，难以稳定生成真实可执行的测试结果。

基于上述分析，本文将 P4-BISG 的设计目标概括为以下四点。

第一，意图驱动。系统应能够直接以用户测试意图为起点，围绕待测网络行为组织测试场景，而不是要求用户先手工指定底层数据包和流表规则。换言之，生成过程应由“测试目标”驱动，而非由“可达路径”驱动。

第二，联合生成。系统应能够联合生成数据包序列、控制平面规则和必要的执行动作，而不是仅输出单一报文。对于许多状态型或策略型程序，仅有数据包而没有与之配套的控制条件，测试场景并不完整。对于包含外部环境变化的程序，仅有报文和规则而缺少执行动作，同样无法构成完整验证过程。

第三，结构化输出。系统最终输出应具有统一结构，并能够明确描述测试中的输入、规则、执行顺序和预期结果。这样生成结果才不仅能被人工阅读，还能直接用于后续回放、验证和实验统计。

第四，稳定可用。由于意图理解和种子生成过程依赖大语言模型，系统必须通过程序上下文约束、多阶段校验和回退机制降低输出不稳定性，保证生成结果具备工程可用性。

基于上述问题定义和设计目标，本文提出 P4-BISG。其核心思想是：以程序上下文为约束基础，以用户测试意图为生成起点，通过多阶段语义规约和结构化生成，将高层测试目标逐步转换为可执行的 P4 测试种子。后续 3.4 节将进一步给出该方法的具体设计。
